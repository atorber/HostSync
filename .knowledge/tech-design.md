# 配置管理工具 — 产品技术设计方案

基于 [PRD](./prd.md)，在现有 HostSync 能力上扩展为「API 管理 + Skill 管理 + 配置管理 + 云端同步」的统一配置管理工具，保持与现有 S3 主机分层同步能力兼容。

---

## 1. 目标与范围

### 1.1 产品目标

- **API 管理**：统一维护 AI 相关 API Key、接入点等配置，支持本地 CRUD 与可选云端同步。
- **Skill 管理**：扫描本地目录发现 `skill`/`skills` 文件夹及其中 `skill.md`，支持增删改查与同步。
- **配置管理**：管理 Claude Code、OpenClaw、Codex 等工具的本地配置文件，复用/扩展现有扫描能力，列表 + 详情编辑。
- **云端同步**：沿用现有 S3 配置（无需额外登录），支持「一键备份」将本地管理的配置与 API/Skill 数据备份到 S3。

### 1.2 兼容性原则

- 不删除、不破坏现有 CLI（`init` / `push` / `pull` / `list` / `scan` / `web`）与 Web UI / 桌面端「主机 + 文件树 + 预览/下载」能力。
- 新功能以「新增模块 + 新菜单/新路由」方式接入，数据存储与 S3 key 命名与现有逻辑可区分、可共存。

---

## 2. 系统架构

### 2.1 整体架构

```
┌─────────────────────────────────────────────────────────────────┐
│                        前端 (React)                               │
│  左侧菜单：API 管理 | Skill 管理 | 配置管理 | [原有] S3 文件浏览     │
│  内容区：列表 + 右侧抽屉详情（新增） / 主机-文件树-预览（保留）      │
└───────────────────────────┬─────────────────────────────────────┘
                             │ invoke / fetch
┌───────────────────────────▼─────────────────────────────────────┐
│                    应用层 (Tauri / Node)                          │
│  • API CRUD  • Skill CRUD + 扫描  • 配置 CRUD + 扫描  • S3 同步   │
└───────────────────────────┬─────────────────────────────────────┘
                             │
        ┌────────────────────┼────────────────────┐
        ▼                    ▼                    ▼
┌───────────────┐   ┌───────────────┐   ┌───────────────────────┐
│ 本地存储       │   │ 本地文件系统   │   │ S3 兼容存储（现有）    │
│ API/Skill 等  │   │ 扫描目录/文件  │   │ 主机分层 + 备份命名空间 │
└───────────────┘   └───────────────┘   └───────────────────────┘
```

- **桌面端**：主入口为 Tauri + React，新功能通过 Tauri Command 访问本地存储与文件系统，S3 沿用现有 Rust 侧实现。
- **Web UI**：可选保留为「仅 S3 浏览」或后续为 API/Skill/配置提供只读或简化能力（需鉴权方案）。
- **CLI**：保留并保留现有命令；新增命令（如 `hostsync api list`、`hostsync skill scan`、`hostsync backup`）与现有 `push`/`pull` 并存。

### 2.2 技术栈

| 层次     | 选型说明 |
|----------|----------|
| 桌面壳   | Tauri 2（已存在） |
| 前端     | React + 现有样式体系，新增列表+抽屉布局 |
| 本地存储 | SQLite（推荐）或 JSON 文件，存 API 配置、Skill 元数据、配置条目索引 |
| 文件访问 | 现有 scanner 扩展 + 新增 Skill 扫描；Tauri 暴露目录选择与读写在权限内 |
| 云端     | 现有 S3 配置与 Rust S3 客户端；备份使用独立前缀与现有主机分层区分 |

## 3. 数据模型与存储

### 3.1 API 管理

- **实体**：每条记录包含：名称、类型（如 openai / anthropic / 自定义）、API Key（加密或脱敏存储）、Base URL（可选）、备注、创建/更新时间。
- **存储**：本地 SQLite 表 `api_keys`，或单文件 `~/.config/hostsync/api-keys.json`（若优先简单实现）。密钥字段建议用本机密钥派生加密后落盘。
- **云端**：备份时写入 S3，例如 `backup/api-keys.json` 或按 host 分片，与现有 `hostname/path/` 命名空间隔离。

### 3.2 Skill 管理

- **实体**：Skill 条目 = 所在目录（绝对路径或规范路径）+ 显示名（可取自目录名或首行标题）+ `skill.md` 内容摘要/全文 + 最后扫描时间。
- **发现规则**：扫描用户指定根目录（可多选），递归查找目录名为 `skill` 或 `skills` 的文件夹，将其下 `skill.md` 视为一个 Skill；同一目录仅一个 `skill.md`。
- **存储**：本地表 `skills` 存路径、名称、内容哈希、摘要等；正文可存表或仍以文件为准、表只存索引与元数据（推荐后者，便于与磁盘一致）。
- **云端**：备份时将 `skill(s)/skill.md` 按相对路径上传，例如 `backup/skills/<path-hash>/skill.md`，避免与现有 push 的 host/path 冲突。

### 3.3 配置管理

- **实体**：与现有 scanner 对齐，每条 = 工具名（Claude Code / OpenClaw / Codex / Cursor 等）+ 范围（项目/家目录）+ 文件路径 + 大小/修改时间；详情即文件内容。
- **发现规则**：复用并扩展 `cli/src/lib/scanner.ts` 的 `PROJECT_TOOLS` / `HOME_TOOLS`；补充 OpenClaw、Codex 等若当前未覆盖的路径。扫描根目录由用户配置（当前目录、工作区根、或指定目录）。
- **存储**：可不持久化到 DB，仅「扫描结果缓存」可选存 SQLite 或 JSON；列表以扫描结果为准，详情按路径实时读文件。
- **云端**：沿用现有 `hostsync push` 语义（按 host + 路径上传）；「一键备份」可对当前扫描到的配置文件批量触发 push 或单独备份命名空间。

### 3.4 云端同步与命名空间

- **S3 配置**：继续使用 `~/.config/hostsync/config.json`，不引入额外登录；桌面端/CLI 共用。
- **命名空间**：
  - 现有：`<hostname>/<path>/...`（主机分层）。
  - 新增备份：使用统一前缀如 `backup/`，其下再分 `api-keys/`、`skills/`、`config-snapshots/` 等，便于与主机分层隔离且便于恢复。
- **一键备份**：将当前本地「API 配置 + Skill 元数据/内容 + 可选配置快照」打包（JSON 或按目录结构）上传到 `backup/` 下带时间戳或版本号的 key；恢复时从 S3 拉取并写回本地。

## 4. 功能模块设计

### 4.1 API 管理模块

- **列表**：分页或全量展示名称、类型、Base URL、更新时间；支持搜索、筛选。
- **详情**：抽屉中展示/编辑名称、类型、API Key（掩码/显密切换）、Base URL、备注；保存写本地存储，可选「同步到云端」写 S3。
- **Tauri**：`api_list`、`api_get`、`api_create`、`api_update`、`api_delete`、`api_export_for_backup`。
- **CLI**（可选）：`hostsync api list`、`hostsync api add`、`hostsync api rm`。

### 4.2 Skill 管理模块

- **扫描**：选择根目录（可多选），后台或同步执行扫描，规则：目录名 `skill` 或 `skills`，其下存在 `skill.md`；结果去重、合并已有 DB 记录。
- **列表**：展示 Skill 名称、路径、最后修改时间；支持按路径/名称搜索。
- **详情**：抽屉中展示/编辑 `skill.md` 内容；保存写回对应文件（若文件被删则提示或转为「仅本地记录」）。
- **Tauri**：`skill_scan_roots`、`skill_list`、`skill_get`、`skill_save`、`skill_delete_record`、`skill_export_for_backup`。
- **CLI**（可选）：`hostsync skill scan [dir]`、`hostsync skill list`。

### 4.3 配置管理模块

- **扫描**：复用并扩展 `scanner.ts`；支持「当前项目」「指定目录」「家目录」等范围；结果列表展示工具名、路径、大小、修改时间。
- **列表**：按工具或按路径筛选；点击行在右侧抽屉展示文件内容（只读或可编辑，若可写则保存回文件）。
- **Tauri**：`config_scan`（传入根目录与 scope）、`config_list`（可带缓存）、`config_read_file`、`config_write_file`（需权限与安全确认）。
- **CLI**：现有 `hostsync scan` 已支持；可扩展 `hostsync config list` 输出 JSON 供 UI 消费。

### 4.4 云端同步模块

- **S3 设置**：桌面端「设置」中展示/编辑 `config.json`（endpoint、bucket、accessKey、secretKey、region、forcePathStyle）；与现有 init 逻辑一致，写回同一文件。
- **一键备份**：将 API 列表、Skill 列表（含内容）、可选配置快照序列化为备份包，上传到 `backup/<timestamp>.json` 或 `backup/<version>/` 下多个对象。
- **恢复**（可选）：从 S3 选择备份版本，下载并解析，写回本地存储与可选文件；需冲突策略（覆盖/跳过/合并）。
- **Tauri**：`backup_run`、`backup_list`（列举 backup/ 下 key）、`backup_restore`；S3 读写复用现有 Rust 实现。

## 5. UI 与交互

### 5.1 布局

- **左侧菜单**：一级菜单固定为「API 管理」「Skill 管理」「配置管理」「S3 文件」（原有主机/文件树/预览）；可折叠。
- **内容区**：  
  - 前三项：顶部可选「扫描」「刷新」；主体为表格/列表；行点击在**右侧抽屉**打开详情（表单或 Markdown 编辑）。  
  - 「S3 文件」：保持现有三栏（主机 | 文件树 | 预览）。
- **设置**：入口在顶栏或侧栏，包含 S3 配置、扫描根目录配置、可选主题等。

### 5.2 列表 + 抽屉规范

- 列表列：根据模块不同见上（API：名称/类型/URL/时间；Skill：名称/路径/时间；配置：工具/路径/大小/时间）。
- 抽屉：标题为当前项名称或路径；内容为表单或编辑器；底部「保存」「取消」；关闭不保存时可选提示。

### 5.3 与现有风格统一

- 沿用现有 `styles.css` 与组件风格（panel、btn、search、list、codebox 等）；新增列表、表单、抽屉使用同一设计语言，保证与 S3 文件浏览页一致。

---

## 6. 安全与权限

- **本地存储**：API Key 建议加密后存；密钥可从本机用户或 Tauri 安全存储派生，避免明文。
- **文件读写**：Skill/配置涉及写盘时，仅在用户明确操作（保存、覆盖）时执行；扫描仅读目录与文件元数据及内容，不写。
- **S3**：继续使用现有 config，不在前端持久化密钥；桌面端通过 Tauri 读 config 与 S3 通信，不暴露到 WebView。
- **备份**：备份包可考虑对敏感字段（如 API Key）加密后再上传，密钥管理可与本地加密一致。

---

## 7. 实施顺序建议

1. **Phase 1 — 数据与后端**  
   - 确定本地存储方案（SQLite vs JSON）并实现 API、Skill 的 CRUD 与存储路径。  
   - 扩展 scanner：增加 Skill 扫描（skill/skills + skill.md）、补齐 OpenClaw/Codex 等配置路径。  
   - Tauri 侧：实现 API 管理、Skill 管理、配置扫描与读写的 commands。

2. **Phase 2 — 前端**  
   - 左侧菜单 + 三个新页面（API / Skill / 配置）的列表 + 右侧抽屉。  
   - 设置页：S3 配置编辑、扫描根目录配置。  
   - 与现有 S3 文件浏览共存，路由或状态区分。

3. **Phase 3 — 云端**  
   - 备份格式与 S3 key 设计（backup/ 前缀）。  
   - 一键备份与可选恢复；CLI 可选 `hostsync backup`。

4. **Phase 4 — 打磨**  
   - API Key 加密、冲突与恢复策略、错误提示与离线体验；CLI 与 Web UI 的增量对齐（若需要）。

## 8. 与现有 HostSync 的对照

| 现有能力           | 处理方式 |
|--------------------|----------|
| CLI init / push / pull / list / scan / web | 全部保留，不删不改语义。 |
| Web UI（Token + 主机/文件树/预览/下载）    | 保留；可改为桌面端一个 Tab 或子路由「S3 文件」。 |
| 桌面端（Tauri + React，S3 同前）          | 保留并作为主入口；新菜单与页面叠加。 |
| config.json（S3）  | 共用；新功能仅增加「设置」中编辑入口。 |
| scanner（AI 配置） | 扩展工具集与 Skill 规则；配置管理直接复用扫描结果。 |
| S3 key 布局        | 主机分层不变；备份使用 `backup/` 前缀，互不覆盖。 |

以上方案在满足 PRD 的前提下，与现有功能兼容、可渐进落地，且为后续「仅本地」与「云端同步」两种使用方式留出清晰边界。
